# 二、什么是Redis发布订阅（Pub/Sub）

Redis Pub/Sub本质：

```
一种实时消息广播机制
```

角色：

```
发布者 Publisher
订阅者 Subscriber
频道 Channel
```

模型：

```
Publisher  ──► Channel ──► 多个 Subscriber
```

特点：

* 实时推送
* 无需轮询
* 多播模式

---

## 2.1 与传统请求响应模型对比

传统HTTP：

```
客户端请求 → 服务器响应
```

Pub/Sub：

```
发布消息 → Redis主动推送
```

本质区别：

```
由拉模型变为推模型
```

---

## 2.2 典型使用场景

适合：

* 实时通知
* 在线用户消息
* 聊天消息广播
* 系统事件通知
* 直播弹幕

不适合：

* 订单处理
* 金融交易
* 可靠任务队列

# 三、Pub/Sub工作原理

Redis内部逻辑：

```
客户端A SUBSCRIBE news
客户端B SUBSCRIBE news
客户端C PUBLISH news hello
```

流程：

```
订阅者注册channel
        │
        ▼
Redis维护订阅关系表
        │
        ▼
发布消息
        │
        ▼
Redis推送给所有订阅者
```

---

## 3.1 内部结构理解

Redis维护：

```
channel -> subscriber列表
```

发布时：

```
遍历订阅者并发送
```

---

## 3.2 最重要的特性：不存储消息

Redis Pub/Sub：

```
不会持久化消息
```

如果：

```
订阅者不在线
```

结果：

```
消息直接丢失
```

这是设计特性，不是缺陷。

# 五、Channel设计规范

错误设计：

```
login
message
data
```

问题：

* 无法扩展
* 无法区分业务
* 易冲突

---

## 5.1 推荐命名规则

建议结构：

```
业务:模块:动作
```

示例：

```
user:login:event
chat:room:message
order:create:event
system:notice:push
```

---

## 5.2 多环境隔离

不建议，实例隔离最好，实在不行dbindex隔离：

```
dev:user:login
test:user:login
prod:user:login
```

# 六、Go实现Pub/Sub模型（核心）

---

## 6.1 发布消息

```go
func Publish(ctx context.Context, rdb *redis.Client, ch string, msg string) error {
	return rdb.Publish(ctx, ch, msg).Err()
}
```

---

## 6.2 创建订阅者

```go
func Subscribe(ctx context.Context, rdb *redis.Client, ch string) {
	sub := rdb.Subscribe(ctx, ch)

	for msg := range sub.Channel() {
		fmt.Println("receive:", msg.Channel, msg.Payload)
	}
}
```

---

## 6.3 为什么必须使用协程

原因：

```
订阅是阻塞行为
```

工程写法：

```go
go Subscribe(ctx, rdb, "order:event")
```

---

## 6.4 完整示例（课堂主代码）

```go
package main

import (
	"context"
	"encoding/json"
	"fmt"
	"github.com/gogf/gf/util/gconv"
	"github.com/redis/go-redis/v9"
	"redis-learn/utils/redisclient"
	"time"
)

func main() {
	ctx := context.Background()
	client := redisclient.GetRedisClient()
	receiveOrderEvent(client, func(msg *EventMsg) error {
		fmt.Println("receiveOrderEvent", msg)
		return nil
	})
	for {
		time.Sleep(time.Second * 1)
		err := sendOrderEvent(ctx, client, &EventMsg{
			OrderID: 1,
			Type:    "create",
			Time:    time.Now().Unix(),
			Msg:     "user 1001 create order",
			UserID:  1001,
		})
		if err != nil {
			fmt.Println("sendOrderEvent err", err)
		}
	}
}

const (
	channelName = "order:event"
)

type EventMsg struct {
	OrderID int    `json:"order_id"`
	Type    string `json:"type"`    // create, update, cancel
	Msg     string `json:"msg"`     // xxx
	Time    int64  `json:"time"`    // create_at
	UserID  int    `json:"user_id"` // 订单user_id
}

func sendOrderEvent(ctx context.Context, client *redis.Client, msg *EventMsg) error {
	return client.Publish(ctx, channelName, gconv.String(msg)).Err()
}

func receiveOrderEvent(client *redis.Client, handle func(*EventMsg) error) {
	go func() {
		ctx := context.Background()
		for {
			sub := client.Subscribe(ctx, channelName)
			message, err := sub.ReceiveMessage(ctx)
			if err != nil {
				fmt.Sprintf("receiveOrderEvent ReceiveMessage %v", err)
				time.Sleep(time.Second * 1)
				continue
			}
			msgStr := message.Payload
			eventMsg := &EventMsg{}
			err = json.Unmarshal([]byte(msgStr), eventMsg)
			if err != nil {
				fmt.Sprintf("receiveOrderEvent Unmarshal %v", err)
				continue
			}
			err = handle(eventMsg)
			if err != nil {
				fmt.Sprintf("receiveOrderEvent handle %v", err)
			}
		}
	}()
}
```

# 八、Pub/Sub vs Stream vs MQ

---

## 8.1 Pub/Sub特点

```
实时推送
无持久化
无确认机制
无重试
```

---

## 8.2 Stream特点

```
可回放
消费者组
可确认
可靠性高
```

---

## 8.3 专业MQ特点（Kafka/RabbitMQ/RocketMQ）

```
高可靠
高吞吐
复杂路由
事务支持
```

---

## 8.4 使用建议

实时通知：

```
Pub/Sub
```

可靠消息：

```
Stream 或 MQ
```
